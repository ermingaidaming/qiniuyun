# 持续交付计划

以下计划用于形成持续的 PR 与 commit 记录。实际排期应按官方批次开始和截止日期调整。

## 第 0 阶段：仓库初始化

- 创建公开评审前可保密的远端仓库。
- 添加 README、PR 模板、提交检查清单。
- 添加可运行的本地 mock demo。

建议 PR：

- `docs: 初始化比赛提交规范文档`
- `feat: 新增本地 mock 口语练习闭环`

## 第 1 阶段：前端实时练习体验

- 麦克风授权与音频电平。
- 练习场景选择。
- 对话流和反馈卡片。
- 移动端适配。

建议 PR：

- `feat: 新增麦克风音量检测`
- `feat: 新增练习场景选择`
- `feat: 新增口语反馈展示`

## 第 2 阶段：后端实时编排基础

- 会话创建与状态管理。
- 实时事件协议。
- Mock 流式返回。
- 后端测试。

建议 PR：

- `feat: 新增练习会话 API`
- `feat: 新增实时事件消息格式`
- `test: 覆盖 mock 反馈生成逻辑`

## 第 3 阶段：AI 能力接入

- VAD 或音频分片。
- 流式 STT。
- LLM 反馈。
- TTS 播放。
- 抢话打断。

建议 PR：

- `feat: 接入语音活动检测`
- `feat: 接入流式语音转写`
- `feat: 新增 LLM 口语反馈链路`
- `feat: 新增 TTS 播放与打断控制`

## 第 4 阶段：评审材料

- README 补齐依赖、原创说明、部署方式。
- 录制 Demo 视频并补充链接。
- 检查所有 PR 描述和 commit 时间。
- 清理密钥、缓存、无授权素材。

建议 PR：

- `docs: 补充 Demo 视频链接与运行说明`
- `docs: 完善依赖声明和原创范围`
- `chore: 提交前清理无关文件`

## 每个 PR 的最低要求

- 只包含一个功能或一类文档改动。
- PR 描述必须包含功能描述、实现思路、测试方式。
- 合并前本地运行通过。
- README 中同步新增第三方依赖。
- 若复用旧代码，PR 描述写明来源。
